01

The 30-second elevator pitch

If you only get one sentence in the interview, this is it.

A user pastes an image URL into a React UI. The Express API validates and enqueues a job. A BullMQ worker downloads the image, runs a sharp transform pipeline (resize, crop, rotate, watermark, encode), and uploads the result to Firebase Storage. The UI receives the result through a Firestore real-time listener — no polling. The whole thing is containerized, observable (Prometheus + structured logs + K8s probes), and hardened for production (API key, rate limit, CSP, graceful shutdown).

What problem does this solve? A "submit URL, get a transformed image back" pipeline that doesn't lock the browser, doesn't lose work on network errors, and scales horizontally by adding more workers.

What makes it production-grade? API key auth, CORS lockdown, per-IP rate limits, hard timeouts, retries with backoff, TTL-based cleanup, Prometheus metrics, K8s probes, multi-strategy anti-bot downloader, OpenAPI 3.0.3 docs.

Why this shape?

  • Browser ↔ Firestore listener — the worker is the only writer, the browser is the only reader. Clean write/read split, no polling, near-instant UI updates.
  • API is stateless — it only validates and enqueues. Adding an API replica is a load-balancer config; no shared state.
  • Worker is stateless — it consumes BullMQ jobs, downloads, transforms, uploads. Add a worker replica → linear throughput. Redis brokers the jobs.
  • Firebase Emulator in dev means zero external dependencies — docker compose up and the whole stack runs locally with the same wire protocol as production.

Why expose worker on no host port?

The worker is a consumer, not a request server. It pulls from Redis. There's nothing for an external client to call on it — exposing a port would just be attack surface.

05

How the functions work

A walkthrough of the four interesting pieces of code, in execution order.

① Request validation — api/jobs.ts

const createJobSchema = z.object({
  url: z.string().url().max(2048)
    .refine(u => {
      const p = new URL(u);
      return p.protocol === "http:" || p.protocol === "https:";
    }, { message: "url must be http(s)" }),
  transform: transformSchema.optional(),
});

One zod schema serves two masters: the runtime validator in jobs.ts AND the OpenAPI spec generator. They cannot drift apart — both read the same source of truth.

② BullMQ enqueue — services/queue.ts + api/jobs.ts

const bullJob = await queue.add(QUEUE_NAME, { url, transform });
const record = { id: bullJob.id, url, status: "pending", /* ... */ };
await createJobRecord(record);   // Firestore write

Why the BullMQ job id == Firestore doc id: the worker only knows the id; it can update the same Firestore document by id alone. No join logic, no transaction.

③ Worker pipeline — workers/imageWorker.ts

// Step 1: download (3 anti-bot strategies)
downloaded = await downloadImage(url);
// Step 2: report progress to Firestore
await updateJobMetadata(id, { bytes, format });
// Step 3: transform — fixed pipeline order
transformed = await transformImage(downloaded.buffer, transform);
// Step 4: upload result
resultUrl = await uploadResult(id, transformed.buffer, mime, fmt);
// Step 5: mark completed
await updateJobRecord(id, { status: "completed", resultUrl, ... });

The transform pipeline order is fixed (resize → rotate → flip → grayscale → crop → watermark → opacity → encode) so the operations compose predictably. Watermark positioning is computed from the post-rotate, post-crop dimensions so the watermark lands exactly where the user clicked even after rotation.

④ The image processor — services/imageProcessor.ts

const { width: postResizeW, height: postResizeH } = computePostResizeDims(
  srcW, srcH, spec.resize,
);
const swapOnRotate = spec.rotation === 90 || spec.rotation === 270;
const preCropW = swapOnRotate ? postResizeH : postResizeW;
const preCropH = swapOnRotate ? postResizeW : postResizeH;

let pipeline: Sharp = applyResize(sharp(input), spec.resize, srcW, srcH);
if (spec.rotation !== 0) pipeline = pipeline.rotate(spec.rotation);
if (spec.grayscale) pipeline = pipeline.grayscale();
// ... crop, watermark (positioned on the post-rotate/crop frame), opacity, encode

Two things worth calling out:

  • Watermark positioning math is done before building the pipeline — it needs the final frame size to compute the 9-zone anchor. We solve for the post-resize/rotate/crop dimensions first, then build the pipeline.
  • Watermark fits-to-image guard: if the overlay is larger than the destination (font size 51 on a 400×400 image), sharp refuses to composite. The fix is to resize the overlay down to fit before compositing, then re-position it.

⑤ The downloader — services/downloader.ts

const strategies = [
  { name: "chrome-windows",  headers: { User-Agent: "Chrome/120.0.0.0...", Sec-Ch-Ua: "...", Referer: "<url host>" } },
  { name: , headers: { User-Agent: "Firefox/121...", // no Sec-Ch-Ua } },
  { name: "safari-macos",     headers: { User-Agent: "Safari/17.2..." } },
];

for (const strategy of strategies) {
  const res = await request(url, { headers: strategy.headers });
  if (BOT_BLOCK.has(res.statusCode)) continue;  // try next
  if (res.statusCode >= 400) throw NOT_FOUND;   // 404 is real
  // ... read stream, return DownloadResult { ..., strategy }
}

Many image CDNs reject a default User-Agent: image-processing-pipeline/1.0 as a bot. Sending real browser headers (User-Agent, Accept, Accept-Language, Accept-Encoding, Sec-Fetch-*, Referer matching the host) clears ~95% of those checks. If one strategy fails with 403/429/503, we try the next. A real 404 is NOT retried (genuine not-found, not a bot block).

06

Production hardening

The four pillars: security, observability, reliability, operational hygiene.

✓ Security

  • API key auth (X-Api-Key, constant-time compare, fail-closed)
  • CORS lockdown to an explicit allow-list
  • Helmet: CSP, HSTS (prod), X-Frame-Options, Referrer-Policy
  • Per-IP rate limit: 30 writes/min, 300 reads/min
  • Request body cap: 64 KB
  • x-powered-by disabled
  • Container: no-new-privileges, non-root user, mem caps

✓ Observability

  • Prometheus /metrics: HTTP histogram, request counter, jobs counter, active gauge, processing duration, image bytes, plus all default Node metrics
  • Structured JSON logs (pino) with per-request requestId
  • X-Request-Id response header on every request
  • K8s-style probes: /health/live, /health/ready
  • OpenAPI 3.0.3 + Swagger UI at /docs

✓ Reliability

  • Per-job hard timeout (120s default)
  • Graceful SIGTERM shutdown — drain in-flight, force-exit after grace
  • TTL-based cleanup — old Firestore docs and Storage objects reaped hourly
  • Cursor-paginated job list (200 max) — bounded payload
  • BullMQ retries with UnrecoverableError for non-retryable failures
  • Auto-scaling watermark overlay to fit destination

✓ Operational hygiene

  • Docker healthchecks on all long-running services
  • Graceful memory caps (mem_limit on each service)
  • Single docker compose up to run the entire stack
  • Health warnings at startup if API_KEY is missing in production
  • RUNBOOK.md for common operator scenarios
07

Design decisions & rationale

The why behind each non-obvious choice. This is what makes the interview go well — the interviewer is going to ask "why this and not that".
Why BullMQ + Redis and not in-process jobs?
decision · queue
Horizontal scale. In-process means a single Node process can only process one job at a time per worker. With BullMQ, the API enqueues, returns immediately, and any number of worker processes consume in parallel. AOF persistence means a Redis restart doesn't lose in-flight jobs. Rejected alternative: pg-boss (Postgres-based) — adds DB pressure; AWS SQS — vendor lock-in.
Why Firestore for state and not a SQL table?
decision · state
Real-time listener is a first-class feature. Polling a SQL row every 500ms is wasteful and adds load. Firestore's onSnapshot is a WebSocket that pushes deltas. The browser subscribes once and just re-renders. We get "real-time updates for free" without writing a single line of pub/sub. Rejected: Postgres + socket.io — more code, more ops.
Why sharp and not ImageMagick?
decision · image processing
Native bindings, no shell-outs. ImageMagick spawns a child process per call — slow, fragile, no streaming. sharp is a libvips binding with lazy evaluation: a chain of operations becomes a single C pipeline executed in one pass. Memory-mapped streaming means we never load the full image into JS. Rejected: Jimp (pure JS, 10× slower), canvas (no formats).
Why the same schema drives validation AND the OpenAPI doc?
decision · type safety
Single source of truth. The zod schemas in api/openapi.ts are used by both the runtime validator (zod's safeParse) and the OpenAPI generator (extendZodWithOpenApi). The docs and the validator cannot drift — when you change a schema, both update. The runtime cost of the .openapi() decorator is zero; it just attaches metadata.
Why a 3-strategy downloader and not just a real browser?
decision · anti-bot
Cost & complexity. 95% of bot blocks are a User-Agent + Accept + Referer mismatch — solvable in undici. A headless browser (Playwright) adds ~300 MB of browser binaries to the worker image, ~5s per request cold start, and is way more code to test. The fallback would slot in as strategy #4 via dynamic import('playwright') if real production needs it.
Why a single Express server and not a Lambda / FaaS?
decision · runtime
Predictable cold-start + image processing > 15s. sharp + a 10 MB image can take 5–30 seconds. Most FaaS platforms charge by the second and have hard time limits. A containerized Express + worker has a fixed 120s budget, scales horizontally, and the worker can be sized exactly for the workload. Rejected: AWS Lambda (15-min timeout but expensive for sustained CPU), Cloud Run (good alternative in production).
Why 3 separate workers for graceful shutdown — not 1 process?
decision · ops
Independent lifecycles. A bug in the image processor doesn't take the API down. Scaling the worker (CPU-heavy) is independent of scaling the API (latency-sensitive). Restart cycles are decoupled. Rejected: single monolith — easier to deploy, but one bad image blocks all submissions.
Why code-split the frontend (React.lazy)?
decision · perf
Bundle size & time-to-interactive. The full bundle was 515 KB — gzipped 141 KB. The Firebase SDK is the biggest dep and only needed once the form is submitted. Splitting it into a lazy-loaded JobList chunk drops the initial bundle to 163 KB (gz 52 KB) and the heavy chunk to 357 KB (gz 90 KB) — but only on demand. TTI on first load is roughly 1/3 of the old value.
Why undici for the HTTP client and not node-fetch or axios?
decision · runtime
It's the default in Node 20. No dependency to install, no version-drift concerns, no shim layer over the underlying connection pool. Built on top of llhttp — one of the fastest HTTP parsers available. We just get connection pooling, HTTP/2, and proper streaming for free.
Why helmet with CSP and not just default headers?
decision · security
Defense in depth. The browser already has a same-origin policy. CSP is a second layer: even if an attacker manages to inject a script (XSS via a future vulnerability), CSP blocks it from loading remote resources or sending data anywhere. The trade-off is more config (we had to relax script-src for Swagger UI) — that's a known cost, not a hidden one.

Talking points for the interview

Pre-canned lines you can drop in when the interviewer probes. Use them to sound confident, not memorized.

When they ask "why this stack?"

  • "I wanted each layer to do one thing well. BullMQ for the queue because it has the best Node ergonomics; Firestore for state because the real-time listener is a free win; sharp because it's the only sane way to process images in Node."
  • "I chose Firebase over a SQL DB specifically because I wanted the browser to subscribe to changes without polling — and that decision saved me a WebSocket server."

When they ask "how does it scale?"

  • "API is stateless — add a load balancer and N replicas. Worker is stateless — add N replicas, the queue load-balances. Redis becomes the bottleneck at >10k in-flight jobs; at that scale you move to Redis Cluster."
  • "The cleanup loop is the only thing that's not horizontally scalable — for a 20-worker fleet you'd move it to a CronJob or use a leader-election lock."

When they ask "what would you do differently?"

  • "I'd add a distributed rate limiter — right now it's in-memory per API replica, so with 3 replicas the user gets 3× the rate limit. Swap in rate-limit-redis with a Redis token bucket."
  • "The downloader is HTTP-only. For real anti-bot at scale I'd add a headless-browser fallback (strategy #4) for the worst offenders."

When they ask "how do you debug prod?"

  • "Every request has an X-Request-Id header. It's in the log line, the response, and any error toast in the UI. I tell the user 'quote the request id' and I can grep one line from prod logs to find the full trace."

When they ask "what's the trickiest part?"

  • "The watermark positioning. Sharp's composite with gravity doesn't have a margin offset, and it trims the SVG render to the tight glyph bounds, so the last character gets clipped. I had to switch to a fixed-size SVG canvas with explicit room for descenders and explicit left/top instead of gravity."
  • "The watermark overflow case: if the watermark is larger than the image, sharp refuses to composite. The fix is to scale the overlay down to fit the destination first, then position it."

When they ask "how do you test?"

  • "76 tests across 6 files. The image-processor tests use sharp to build a tiny test PNG, run the transform, then re-read the output and assert dimensions/format. The downloader tests mock undici with a fake response. The OpenAPI doc has its own test that verifies all paths and schemas are registered."
  • "I leaned toward contract tests over unit tests — assert the public shape (the OpenAPI spec, the JobRecord JSON), not the internals."

When they ask "how do you ship this?"

  • "One docker compose up --build brings up all 5 services. CI runs typecheck + tests on every PR. The README has a hardening checklist for production: real API key, real Firebase project, ALLOWED_ORIGINS, TLS terminator in front of the API."
